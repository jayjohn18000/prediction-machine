## Steps (in execution order)

### Step 1 — Sport Inference Backfill
**What:** Re-run `sport-inference.mjs` across all 44K `unknown` sports markets.
**Why:** The proposal engine needs sport labels to block cross-sport false matches (NFL vs NBA, etc.). Without this, proposals are either garbage or suppressed by the semantic guard.
**How:** Batch SQL update using the existing `inferSportFromKalshiTicker()` and title-based inference for Polymarket. Run as a one-shot script.
**Exit:** `unknown_sport < 500` (down from 1,663+).

### Step 2 — Stale-Active Cleanup
**What:** Identify and resolve the 8,317 `stale_active` markets.
**Why:** These are markets marked active but not receiving snapshots. They pollute the proposal universe and inflate false match rates.
**How:** Run the inactive-guard check (CLAUDE.md invariant). Markets past `close_time` get status flipped to `closed`. Markets with no snapshot in 7+ days and no close_time get flagged for manual review.
**Exit:** `stale_active < 100`.

### Step 3 — Semantic Violation Cleanup
**What:** Resolve the 369 semantic violations from the latest audit.
**Why:** Violations block the strict-audit gate, which blocks E2. Also means existing proposals may be low quality.
**How:** Run `pmci:audit:sports:packet`, categorize violations (cross-sport? title mismatch? stale family?), batch-fix or deactivate.
**Exit:** `semantic_violations = 0`.

### Step 4 — Sports Proposal Engine Full Run
**What:** Run `runProposalEngine({ category: 'sports' })` on the cleaned universe.
**Why:** This is where matched pairs come from. The last run considered 12.3M pairs and inserted 66 — with clean sport labels and no stale markets, the acceptance rate should improve significantly.
**How:** Run proposer, then run strict-audit packet. Review auto-accepted vs. pending proposals. Target: 200+ equivalent pairs accepted.
**Exit:** `current_links (sports) >= 200` with `semantic_violations = 0`.

### Step 5 — Enable Observer DB Discovery
**What:** Set `OBSERVER_DB_DISCOVERY=1` in production `.env`.
**Why:** The observer currently only watches static pairs from JSON config. DB discovery mode (merged today) queries `pmci.market_links` for active equivalent pairs and adds them to the observer cycle automatically.
**How:** Add env var, restart observer, verify new pairs appear in cycle logs.
**Exit:** Observer logs show `Active pairs: N (M static + K DB-discovered)` with K > 0. Snapshots flowing for linked sports pairs on both sides.

### Step 6 — Competitive Baseline Research
**What:** Pull data from OddPool, OddsJam, or similar arb infrastructure for the same events PMCI is tracking, and compare.
**Why:** Need to know: (a) how many cross-platform pairs they surface for the same sports, (b) what spread detection latency looks like, (c) what their fee-adjusted edge thresholds are. This sets the coverage and latency targets for Phase F.
**How:** Manual research + API exploration. Document findings in `docs/competitive-baseline.md`.

**Key sources to evaluate:**
- **OddPool** — prediction market arb scanner (Kalshi/Polymarket focused)
- **OddsJam** — sports betting arb finder (broader, but has prediction market support)
- **Polymarket Whales / Kalshi arb bots** — Twitter/Discord community signals for what active arb traders are doing
- **Academic/blog references** — fee structures, typical spread ranges, opportunity persistence data

**Questions to answer:**
1. How many Kalshi↔Polymarket sports pairs do competitors track? (Our target: match or exceed)
2. What is their snapshot/refresh interval? (Ours: currently 4hr scheduled + observer cycle)
3. What spread threshold do they flag as actionable? (Informs F1 tradability model)
4. What are actual Kalshi/Polymarket fee structures for sports markets? (Needed for net-edge calc)
5. How long do sports arb opportunities typically persist? (Edge half-life — informs F2)

**Exit:** `docs/competitive-baseline.md` written with quantitative answers to the 5 questions above.

### Step 7 — Spread Dashboard / Signal Validation
**What:** With 200+ linked pairs receiving live snapshots, generate the first real spread analysis.
**Why:** This is the deliverable that proves the system is producing actionable intelligence. It also validates whether the data quality is good enough for Phase F.
**How:** Query `pmci.provider_market_snapshots` for linked sports pairs, compute time-series spreads, identify the largest and most persistent divergences.
**Output:** Either a one-shot report or a simple dashboard (could be a Lovable app or HTML artifact) showing:
- Top 20 sports spreads by magnitude
- Spread persistence (how long did each opportunity last?)
- Coverage comparison vs. competitive baseline from Step 6

**Exit:** At least 10 sports pairs show measurable, non-zero spreads with bilateral price data.

Without E1.6, Phase F is a planning exercise with no data to model against, and E2 remains blocked.
